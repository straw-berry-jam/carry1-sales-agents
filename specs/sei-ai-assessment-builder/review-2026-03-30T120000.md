## Security Check (business terms)

- **Who can touch an assessment?** Today, routes scope rows by a **fixed stub user ID**. That means “security” equals **who can reach your Vercel app and guess IDs**. For a **private internal** tool behind existing access control, this is often acceptable for MVP. For **anything customer-facing or internet-exposed without VPN**, this is **not** acceptable — anyone who can call the API could manipulate another assessment if they discover UUIDs.

- **Uploaded files**: Extraction and storage paths should stay aligned with upload limits (covered by unit tests). **Malicious file content** is a known class of risk; MVP scope usually accepts “trusted internal users.”

- **Debug logging**: Several `console.log` calls print **previews of draft HTML** (including findings) on the **client and server**. That can put **client names or engagement details** into browser consoles and server logs — fine for debugging, **not ideal for production** under strict data-handling rules.

**Summary**: Acceptable for a **locked-down internal** pilot if the team removes or gates debug logs before production and plans **real user identity** on assessments.

## Concerns (should address)

1. **Debug `console.log` in publish, save-draft, export-docx, and the workspace**  
   - **Why it matters**: Clutters Vercel logs and can leak **sensitive draft text** into log sinks.  
   - **Fix**: Remove or wrap in `if (process.env.NODE_ENV === 'development')` (or a dedicated `DEBUG_ASSESSMENT_BUILDER` flag).  
   - **Time**: ~30–60 minutes.

2. **Stub user**  
   - **Why it matters**: No per-user isolation until real auth lands.  
   - **Fix**: Tracked as follow-up (already noted in code TODO).
