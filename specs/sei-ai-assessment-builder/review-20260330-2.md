## Executive summary (for merge decisions)

The branch adds a large, coherent **Assessment Builder** slice: dashboard, forms, **extract → generate → refine** pipeline, **contenteditable** workspace with SEI Guide chat, **debounced save**, and a **Publish Draft** API that runs in a **single database transaction** and accepts the **live draft from the request body** (not only auto-saved state). **Unit tests** pass (46 tests across 13 suites).

**Gap vs Phase 4 / full P1 spec**: After publish, the app **navigates to `/guide/assessment-builder/[id]/published`**, but that **route is not implemented** in this branch — users would see a **404**. There is **no** published read-only view, **version history UI**, **restore**, **DOCX export**, or **success toast** yet. **`assessment_versions.summary`** is not in the Prisma schema; publish does not write a summary line.

**Recommendation**: **Do not merge as “Phase 4 complete.”** Either **add the published page** (even a stub) before merge, **or** change navigation to stay on the builder until publish + published ship, **or** merge with explicit scope that **P1 end-to-end is not claimed**.

---

## What’s working well

- **Publish saves the right thing**: The client reads **current section HTML** from the editor DOM and sends it as `draft`; the server updates `draft_content` and status in the **same transaction** as the version row. That matches the “don’t trust auto-save alone” requirement.
- **Version numbers**: `nextPublishVersion` is tested (`v1.0` → `v1.1`, etc.) and used in publish.
- **Internal consistency**: Builder routes, `lib/assessment-builder-*` modules, and stub user scoping are aligned with the constitution’s **Decision 7** (no auth on these routes in v1).
- **Tests**: Chunking, draft schema, dashboard, uploads, extract text, versioning, and embeddings (mocked) are covered.
