## What's Working Well

- **Clear separation of concerns** — Scoring prompts live in one place; the API only validates input, fetches the coach prompt from the database, and calls Claude. Easy for future developers to change calibration or add session types.
- **Admin control is real** — The system prompt for scoring always comes from the database (Prompt Control). Changing the SPIN Sales Coach prompt in Admin changes scoring behavior with no code deploy.
- **Predictable errors for integrators** — Missing or invalid input returns 400 with a clear message; no active agent returns 404 with the exact spec message so the UI or docs can tell users to set the agent to active; parse or API failures return 500 with safe, non-leaky messages.
- **Markdown stripping** — The route strips ```json/``` fences before parsing, so scorecard JSON is returned even when the model wraps it in markdown.
- **Empty or weak agent prompt** — If the agent’s prompt is null or only whitespace, the code treats it as “no active agent” and returns 404, which matches the spec.

## Test Coverage

**What would be tested (manual today)**:

- Happy path: POST with valid `transcript` and `sessionType` and an active SPIN Sales Coach → 200 and JSON scorecard with expected shape.
- Error cases: Missing `transcript` or `sessionType` → 400; invalid `sessionType` → 400; no active agent → 404 with exact message; invalid JSON from model or Anthropic failure → 500.
- Edge cases: Empty string transcript allowed; sessionType restricted to the four allowed values.

**Automated tests**: None (no test runner in repo). Coverage N/A.

**Recommendation**: When a test framework is added, add tests for: (1) validation (400 cases), (2) 404 when `getActiveSpinCoachPrompt` returns null, (3) markdown stripping and JSON parse, (4) that the route does not hardcode a system prompt.

## Security Check

- **Input validation**: Request body is validated; `transcript` and `sessionType` are required; `sessionType` must be one of four allowed values. Prevents malformed or unexpected input from driving the scorer.
- **Error messages**: 400/404/500 responses use generic or spec-defined text only. No user-supplied transcript or internal details are echoed, so no information leakage.
- **Secrets**: Anthropic API key is read from environment; not logged or returned.
- **Potential abuse**: Very long transcripts could consume a lot of tokens and cost. The spec does not require a length limit; acceptable for MVP. If cost or abuse becomes an issue later, add a reasonable transcript length cap (e.g. 50k characters) and return 400 when exceeded.

---

## Performance Check

- **Database**: One Supabase query per request (active SPIN Sales Coach prompt). Simple, indexed-friendly filter (name + status); no N+1 or heavy queries.
- **Anthropic**: One non-streaming call per request. Latency is dominated by the model; no extra round-trips.
- **Scalability**: Suitable for on-demand scoring (e.g. after a session). If traffic grows, consider rate limiting or a queue; not required for current scope.
