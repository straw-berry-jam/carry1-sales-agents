# Bootstrap Command

Guide a non-technical founder through setting up a new project and generating a complete CLAUDE.md constitution. Follow **Non-Technical Founder Mode** from CLAUDE.md: zero jargon, business-focused questions, specific recommendations.

---

## Phase 1: Interactive Interview

**If the user runs `/bootstrap` with no answers** (e.g., just `/bootstrap`): Ask all questions below in one message. Organize by category. Tell the user: *"Reply with your answers — you can use bullet points, paragraphs, or answer category by category. I'll use sensible defaults for anything you skip."*

**If the user runs `/bootstrap` and includes answers** (e.g., `/bootstrap` followed by answers, or answers in the same message): Skip to Phase 2 and proceed with generation.

### Project Basics
- What does your app do? (in one sentence)
- Who are the users? (customers, internal team, partners, etc.)
- What's the #1 problem this solves?

### Platform & Scale
- Do you need: Web app, Mobile app (iOS/Android), or Both?
- Expected users in Year 1? (< 100, 100-1k, 1k-10k, 10k+)
- Any specific requirements? (works offline, real-time updates, etc.)

### Integrations & Data
- Does it handle payments? (Stripe, other, no)
- Does it send emails/notifications? (yes/no)
- Does it integrate with other tools? (list them)
- What type of data does it store? (user accounts, documents, analytics, etc.)

### Business Context
- Timeline pressure? (MVP in weeks, stable product in months, no rush)
- Team situation? (solo founder, planning to hire, have co-founder)
- Budget constraints? (bootstrap/free tools preferred, can pay for best tools)

### Branding & Design Guidelines
- Do you have any branding or design guidelines? (colors, logo, fonts, tone, existing style guide, etc.)
- If yes: Share them (paste, describe, or link). These will be added to CLAUDE.md so the /design command and implementation stay consistent.

### Linear Project Setup
- Would you like to set up a new project in Linear for this app? (yes/no)
- If yes: What should the Linear project be named? *(Default suggestion: match the Cursor workspace name — e.g., if the workspace folder is `my-app`, suggest "My App")*

## Phase 2: Technical Decisions (AI Makes All)

When the user provides answers, **parse them** and **make all technical decisions**. Do NOT ask technical questions. Use this decision logic:

### Tech Stack Decision Matrix

| Answer Signal | Recommendation | Business Reasoning |
|---------------|----------------|-------------------|
| Web only, MVP fast, bootstrap budget | Next.js + PostgreSQL + Vercel | Fast to build (4-6 weeks), free tier, industry standard for hiring |
| Web only, complex business logic, forms/workflows | Rails + PostgreSQL + Heroku/Render | Best for CRUD-heavy apps, mature ecosystem, fast iteration |
| Web + Mobile needed | Next.js (web) + Expo (mobile) | Share logic, one codebase for mobile, React skills transfer |
| Real-time updates needed | Add Supabase or Pusher | Real-time without building from scratch |
| Payments (Stripe) | Stripe SDK + webhooks | Industry standard, 2.9% + 30¢, handles compliance |
| Emails/notifications | Resend or SendGrid | Reliable, free tier, easy setup |
| Year 1 < 1k users | Vercel/Render free tier | $0-20/month |
| Year 1 1k-10k users | Vercel Pro or Render Starter | ~$20-50/month |
| Year 1 10k+ users | Scale plan, consider caching (Redis) | Plan for growth |
| Solo founder | Next.js or Rails (pick one) | Fewer context switches, huge community |
| Planning to hire | Match to common stack (Next.js, Rails) | Easier to find developers |
| Works offline | PWA + service workers, or Expo offline | Add later; not MVP-critical |
| Bootstrap/free preferred | Vercel, Supabase, Resend free tiers | Minimize cost until revenue |
| Can pay for tools | Use best-in-class (Vercel Pro, etc.) | Worth it for reliability |

### Decision Output Format

For each major choice, state:
- **I recommend [X]** because: [1-2 sentences in business terms — time, cost, risk, hiring]
- Example: "I recommend Next.js + PostgreSQL + Vercel because: Fast to build (4-6 weeks for MVP), industry standard (easy to hire for later), scales to 100k users, hosting costs ~$20/month."

## Phase 3: Generate CLAUDE.md

Replace the template [CLAUDE.md](CLAUDE.md) with a **complete, project-specific** constitution. Fill in ALL placeholders:

1. **Title**: `# [Project Name] Project Constitution` — Use the app name from answers
2. **Remove** the "How to Customize" section (replace with: "This constitution was generated by `/bootstrap`. See [bootstrap-summary.md](bootstrap-summary.md) for the interview and decisions.")
3. **Core Principles I–IV**: Keep Non-Technical Founder Mode, SDD, Linear, TDD as-is
4. **Core Principles V–VI**: Replace placeholders with project-specific principles (e.g., "Stripe Integration", "Email Delivery", "Multi-tenancy" — only if relevant)
5. **Domain Model**: Create from user's description — core entities, flows, key terms
6. **Code Style**: Fill with stack-specific conventions (e.g., ESLint + Prettier for Next.js, RuboCop for Rails)
7. **Architectural Constraints**: Match chosen architecture
8. **Tech Stack Anchoring**: Fill table with chosen stack and versions
9. **Directory Contract**: Fill based on stack (e.g., Next.js: `src/app/`, `src/components/`, `__tests__/`)
10. **Brand & Design Guidelines**: If the user provided branding or design guidelines, add a new section to CLAUDE.md (e.g., "Brand & Design Guidelines") with their colors, fonts, tone, logo notes, or style guide. This ensures /design and implementation stay consistent. If none provided, add: "*(To be filled when branding is defined.)*"
11. **Gotchas**: Leave as "*(To be filled as issues are discovered during development.)*"
12. **Version**: Use today's date

## Phase 4: Create Project Structure

Ensure these exist (create if missing):

```
specs/
├── templates/
│   ├── FEATURE_TEMPLATE.md      # Copy from starter kit if missing
│   └── ARCHITECTURE_TEMPLATE.md # Copy from starter kit if missing
├── features/                    # Empty or existing specs
├── bugs/                        # Bug reports (from /debug)
├── BUG-REGISTRY.md              # Master bug list (create if missing)
└── architecture/                # Empty, ready for ADRs
```

- **.gitignore**: Ensure it includes `.mcp.json` (contains API keys). Add common entries: `node_modules/`, `.env`, `.env.local`, `dist/`, `build/`, etc. based on stack.
- **README.md**: Update with:
  - Project name and one-sentence description
  - Link to [GUIDE.md](GUIDE.md)
  - Project structure
  - Quick start (if applicable)
  - "Generated by `/bootstrap`"
  - **If Next.js**: Include Local Development section with `dev:safe`, `verify:dev`, `build:clean`, `verify:build`, `ensure:build`, and scripts (`reset-dev-server.sh`, `fix-webpack-error.sh`, `verify-build-health.sh`, `ensure-build-health.sh`). See [docs/NEXTJS-DEV-BUILD-SETUP.md](docs/NEXTJS-DEV-BUILD-SETUP.md).

### Linear Project (if user said yes)

If the user wants a new Linear project:

1. **Determine project name**: Use the name they provided, or default to the Cursor workspace name (e.g., folder `my-app` → "My App", `acme-dashboard` → "Acme Dashboard"). Convert to title case.
2. **Create the project**: Use Linear MCP `create_project` with the project name and the user's default team (from Linear MCP context).
3. **Include in output**: Add the Linear project URL to the summary and bootstrap-summary.md.

If Linear MCP is not available, inform the user: "Linear MCP is not connected — create the project manually in Linear and link it when ready."

## Execution Flow

1. **First message**: User runs `/bootstrap` → AI asks all interview questions (Phase 1), including Linear project setup
2. **Second message**: User provides answers → AI executes Phases 2–6: makes decisions, generates CLAUDE.md, ensures project structure, creates Linear project (if requested), outputs summary, saves bootstrap-summary.md
3. **If answers are incomplete**: Use sensible defaults, note assumptions in bootstrap-summary.md
4. **Linear project name**: If user doesn't specify a name, use the Cursor workspace/folder name as the default suggestion (e.g., `my-app` → "My App")

## Timeline Estimation Guide

| Signals | MVP Estimate |
|---------|---------------|
| Simple CRUD, web only, no payments | 3-4 weeks |
| Auth + CRUD + 1-2 integrations | 4-6 weeks |
| Auth + payments + emails + complex flows | 6-10 weeks |
| Mobile + web, real-time | 8-12 weeks |

## Do Not

- Ask technical questions ("PostgreSQL or MongoDB?")
- Present open-ended choices without a recommendation
- Use unexplained jargon in the interview or output
- Skip filling any CLAUDE.md placeholder (use "TBD" only if truly unknown, with a note to fill later)
